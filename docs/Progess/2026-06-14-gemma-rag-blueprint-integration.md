# Gemma/RAG Blueprint Integration

## Purpose

SmartFarm Blueprint generation now uses the current twin state as the decision input and TwinX Gemma/RAG as a grounded setpoint/evidence provider.

The design intentionally keeps responsibilities separated:

```text
current sensor/crop/actuator/vision snapshot
  -> TwinX rag-strawberry /recommend
  -> SmartFarm gap analysis
  -> Gemma/RAG candidate actuator recipes
  -> Omniverse Twin simulation + scoring
  -> runtime blueprint registration + Apply
```

Gemma/RAG does **not** directly decide the final actuator plan. It provides literature-grounded strawberry setpoints and explanatory evidence. The SmartFarm Twin compares those setpoints with the current virtual sensor/crop state, generates candidates, simulates each candidate, and selects the highest-scoring result.

## Runtime endpoints

Twin extension:

- `POST /smartfarm/blueprint/generate`
- `POST /smartfarm/planning/run` with `provider: "gemma-rag"`
- `POST /smartfarm/blueprint/apply`

OmniOps UI:

- `Generate Gemma/RAG Blueprints`
- `Apply Recommended`
- Evidence dashboard shows RAG provider, growth stage, source count, gap factors, candidate rationale, and simulation scores.

## Configuration

Spark node runtime uses environment variables exported by `scripts/smartfarm-twin-common.sh`:

```bash
SMARTFARM_RAG_BASE_URL=http://10.38.38.241:8080
SMARTFARM_RAG_TOKEN_FILE=$HOME/.smartfarm-rag-token
SMARTFARM_RAG_TIMEOUT=30
```

The token file is intentionally outside the repository.

TwinX GitOps exposes `rag-strawberry` as an internal LoadBalancer on `10.38.38.241:8080` with Bearer-token authentication still enabled.

## Candidate contract

Generated candidates are runtime-registered/displayed as exactly three operator choices:

- `blueprint-a` / `Plan A`
- `blueprint-b` / `Plan B`
- `blueprint-c` / `Plan C`

The UI intentionally hides strategy suffixes such as low-cost/balanced/disease-safe. Those details stay inside each plan explanation/tradeoff row instead of the plan label.

Each candidate includes:

- `actuatorTarget`
- `sensorTarget`
- `expectedSensorShift`
- `ragEvidence`
- `gapAnalysis`
- `simulation.dailyStates`
- `predicted.shipmentDate/yieldScore/opexDeltaPercent/diseaseRisk`

After generation, Gemma/RAG dynamic candidates are inserted into:

- `BLUEPRINT_SENSOR_STATES`
- `BLUEPRINT_ACTUATOR_STATES`
- `BLUEPRINT_SERVICE_SUMMARY`
- `latestPlanningRun`

`baseline` is intentionally **not** overwritten in those runtime registries. It remains the immutable current-operation/reset anchor (Day 34 flowering/delayed fruit-set state), while the planning-run baseline comparison stays inside `latestPlanningRun`. This prevents Apply/Reset Baseline from drifting to a prior projected harvest-day state.

That makes generated blueprints apply through the same path as static Plan A/B/C without corrupting the demo starting state.

## Operational notes

- If `SMARTFARM_RAG_BASE_URL` is missing or RAG fails, `Generate` falls back to the deterministic planner and marks `gemmaRagStatus` as unavailable.
- Do not store RAG/Gemma tokens in git.
- Keep `rag-strawberry` token-protected even though it is only exposed on the internal LAN.
### 2026-06-15 UI text safety update

The TwinX Gemma/RAG request now includes an English ASCII UI text contract. If the endpoint still returns Korean or other non-ASCII human-facing plan text, the adapter keeps numeric actuator targets but replaces dashboard-facing rationale/intent/tradeoff/control text with deterministic English fallback copy. This prevents Omniverse UI from rendering unsupported glyphs as `???` in Plan A/B/C cards.

### 2026-06-15 classroom verification trace

The TwinX RAG adapter now records the last live request trace (`method`, `path`, `url`, HTTP status/error, response keys, and a compact body summary with objective, candidate count, current day, planting date, vision attachment, and key sensor values). OmniOps surfaces that evidence in a new selectable bottom tab, `SmartFarm RAG Trace`, and appends the same terminal-style lines to `logs/smartfarm-blueprints/rag-trace.log`.

This makes a Generate click auditable during a live class demo:

1. `Generate Gemma/RAG Blueprints` logs the Twin generate request.
2. The Twin calls `/blueprints/generate` on the configured TwinX Gemma/RAG endpoint, or records the legacy/fallback status.
3. The panel shows source count, gap factors, Plan A/B/C scores, ship deltas, and the saved JSON trace path.
